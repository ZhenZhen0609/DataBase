二、核心代码文件详解

2.1 main.cpp - 程序入口
功能：Qt应用程序的主入口点，负责初始化应用并根据配置决定启动GUI或运行控制台测试。
当前实现：
- 通过静态常量 RUN_TESTS_ONLY 控制程序走向。
- 若 RUN_TESTS_ONLY = true：创建 QCoreApplication，依次运行 AuthManager 注册/登录逻辑、SchemaManager 字段校验与表结构测试、RecordManager 数据插入与查询测试，最后退出。
- 若 RUN_TESTS_ONLY = false：创建 QApplication，显示 MainWindow 进入标准 GUI 事件循环。
设计原理：
- 利用条件变量控制入口，方便开发者在"调试阶段验证后端逻辑"与"集成阶段展示界面"之间灵活切换，无需修改多个文件。

2.2 mainwindow.cpp/h - 主窗口（粉圈D）
功能：应用程序的主界面窗口，提供完整的数据库管理UI交互。
核心功能（阶段一）：
- 登录/注册界面
- 数据库创建与删除
- 左侧QTreeView显示数据库树状结构

核心功能（阶段二）：
- 建表对话框（动态添加字段）
- 表结构展示（QTableView）
- 数据浏览（QTableView）
- UI联动刷新（建表后自动刷新树状图）

核心功能（阶段三）：
- 右键菜单（数据库：新建表/删除库；表：删除表）
- 搜索工具条（支持文本字段模糊搜索）
- 全局日志控制台（带时间戳）
- 插入记录弹窗（根据表结构动态生成表单）

设计原理：
- 使用QMainWindow作为基类，提供菜单栏、工具栏等标准组件
- 通过QTreeWidget实现数据库/表的树状展示
- 使用QTableWidget展示表结构和数据
- 通过信号槽机制实现UI联动
- 使用QMenu实现右键上下文菜单

2.3 common.h - 公共类型定义
功能：定义项目中全局使用的数据类型和常量。
主要内容：
- FieldType枚举：支持的数据类型（INT、TEXT、DOUBLE、BOOLEAN）
- Field结构体：字段定义（名称、类型、长度、是否必填、是否主键）
- TableSchema结构体：表结构定义（表名、字段列表）
- TableBlock结构体：表描述块（表名、记录数、字段数、文件路径、时间戳）
- ResponseStatus枚举：操作返回状态（OK、ERROR、DB_NOT_FOUND等）
- Response结构体：统一响应格式（状态、消息、数据）
- Config命名空间：全局常量（DATA_PATH = "./data/"、USER_FILE = "users.json"）
设计原理：
- 使用命名空间管理全局常量，避免命名冲突
- 枚举类型提供类型安全的状态标识
- 结构体封装相关数据，便于传递和管理

2.4 storagemanager.cpp/h - 存储管理器（红圈A）
功能：负责数据库文件夹和表物理文件的创建、管理、修改和删除。
核心方法（阶段一）：
- createDatabase(username, dbName)：创建数据库文件夹
- dropDatabase(username, dbName)：删除数据库文件夹及所有文件

核心方法（阶段二）：
- createTable(username, dbName, tableName)：创建空表文件（兼容旧接口）
- createTable(username, dbName, tableName, fields)：创建表并写入字段结构到.tdf文件
- loadTableSchema(username, dbName, tableName)：从.tdf文件读取表结构
- dropTable(username, dbName, tableName)：删除表的.tdf和.trd文件

核心方法（阶段三）：
- alterTable(username, dbName, tableName, newFields)：修改表结构（添加/删除/修改字段）
- writeLog(username, dbName, const QString &action)：写入操作日志到ruanko.log

实现逻辑（createTable带字段参数版本）：
1. 确认数据库文件夹存在
2. 构建文件路径：{username}/{dbName}/{tableName}.tdf、{username}/{dbName}/{tableName}.trd、{username}/{dbName}/{dbName}.tb
3. 检查表是否已存在
4. 创建.trd数据记录文件（空文件）
5. 组装TableBlock头部信息（表名、记录数、字段数、文件路径、时间戳）
6. 将TableBlock写入.tb文件（二进制格式）
7. 将字段结构（名称、类型、长度、是否主键等）以JSON格式写入.tdf文件
8. 写入操作日志到ruanko.log
9. 输出成功日志：[Storage] Table xxx created in xxx with x fields.

设计原理：
- 使用Qt的QDir类进行文件系统操作
- 使用QJsonDocument将表结构序列化为JSON格式存储
- TableBlock使用二进制格式写入.tb文件，提高读取效率
- 操作日志系统记录所有创建和删除动作，便于追踪和审计

2.5 AuthManager.cpp/h - 认证管理器（橙圈B）
功能：负责用户登录验证、新用户注册及密码安全管理。
核心方法：
- login(user, pwd)：验证用户名和密码（密码通过SHA256哈希比对）。
- registerUser(user, pwd)：注册新用户，密码加密后写入文件。
- hashPassword(pwd)：对明文密码进行SHA256哈希加密。
- loadUsers()：从JSON文件加载用户数据。
- findUserFile()：多路径智能搜索users.json位置。
- createDefaultUserFile()：若文件不存在，自动创建包含默认账号(admin/123456)的初始文件。
登录验证流程：
用户输入 → login() → loadUsers() → findUserFile() → 读取JSON → 哈希比对
设计原理：
- 使用QFile和QJsonDocument进行数据持久化。
- 密码使用QCryptographicHash::Sha256加密，避免明文存储风险。
- 支持多路径搜索（优先当前/可执行目录，其次上级/根目录），适应不同运行环境。

2.6 SchemaManager.cpp/h - 模式管理器（橙圈B）
功能：负责数据库字段（模式）的类型校验和表结构持久化管理，与StorageManager协作完成物理文件操作。
核心方法（阶段一）：
- validateFieldType(type, value)：校验某个值是否符合指定的类型约束（支持INT、TEXT、DOUBLE、BOOLEAN）
- validateField(field, value)：校验某个值是否符合指定的字段定义（包含必填检查）
- validateRecord(schema, record)：校验整条记录的数据类型是否符合表结构

核心方法（阶段二）：
- createTable(username, dbName, schema)：调用StorageManager创建表物理文件并写入字段定义
- loadTableSchema(username, dbName, tableName)：调用StorageManager从.tdf文件加载表结构
- loadTables(username, dbName)：扫描数据库目录下所有.tdf文件，加载所有表结构
- dropTable(username, dbName, tableName)：删除表的.tdf和.trd物理文件

核心方法（阶段三）：
- validateAndFillRecord(schema, existingRecords, record)：数据类型强校验、主键唯一性检查、默认值填充
- alterTable(username, dbName, tableName, newFields)：修改表结构（添加/删除/修改字段）

设计原理：
- SchemaManager负责业务逻辑层面的表结构管理
- StorageManager负责底层物理文件操作
- 通过协作模式实现职责分离，提高代码可维护性
- 数据校验采用多层校验机制：字段类型校验 → 主键唯一性校验 → 默认值填充

2.7 RecordManager.cpp/h - 记录管理器（蓝圈C）
功能：负责数据记录的增删改查操作，支持结构化数据校验和二进制存储。
核心方法（阶段一）：
- insertRecord(username, dbName, tableName, data)：插入记录（旧版.json格式）
- selectAllRecords(username, dbName, tableName)：查询所有记录（旧版.json格式）

核心方法（阶段二）：
- insertRecord(username, dbName, tableName, data)：插入前读取.tdf校验字段类型，序列化写入.trd文件
- selectAllRecords(username, dbName, tableName)：从.trd文件反序列化读取所有记录
- loadTableSchema(username, dbName, tableName)：从.tdf文件加载字段定义
- validateRecord(fields, data)：校验记录数据类型是否符合字段定义
- serializeRecord(record, fields)：将记录序列化为二进制格式
- deserializeRecord(data, fields)：从二进制数据反序列化记录

核心方法（阶段三）：
- updateRecord(username, dbName, tableName, recordId, newData)：根据主键更新记录
- deleteRecord(username, dbName, tableName, recordId)：根据主键删除记录
- selectWhere(username, dbName, tableName, fieldName, value)：条件查询（支持模糊匹配）

实现逻辑（阶段三完整版本）：
1. 插入：读取.tdf校验字段类型 → 校验主键唯一性 → 序列化 → 写入.trd
2. 查询：读取.trd文件 → 反序列化 → 返回QJsonArray
3. 更新：遍历.trd找到匹配记录 → 替换数据 → 重写文件
4. 删除：遍历.trd过滤掉匹配记录 → 重写文件
5. 条件查询：遍历.trd → 字段匹配 → 返回匹配记录

设计原理：
- 使用QDataStream进行二进制序列化，提高存储和读取效率
- 每条记录自动添加_created_at字段记录创建时间
- 支持INT、TEXT、DOUBLE、BOOLEAN四种数据类型的强校验
- 返回统一的Response结构体，包含状态、消息和数据

2.8 users.json - 用户数据文件
功能：存储用户账号和加密后的密码哈希值。
内容格式：
{
    "admin": "8d969eef6ecad3c29a3a629280e686cf0c3f5d5a86aff3ca12020c923adc6c92",
    "user1": "f561aaf6ef0bf48046a1449572e684c6348e9e0a47404e440814b4264a02319d"
}
存储位置：
- 由 AuthManager 自动搜索路径决定，通常位于可执行文件同目录下。

2.9 CMakeLists.txt - 项目构建配置
功能：CMake构建系统的配置文件，定义项目编译规则。
主要配置项：
- 指定CMake最低版本要求(3.16)
- 设置项目名称和版本
- 启用Qt自动编译特性(AUTOUIC、AUTOMOC、AUTORCC)
- 添加源文件到项目（包括新增的 SchemaManager 和 RecordManager 文件）
- 链接Qt库（Widgets、Core）

三、数据流向图（阶段三完整版本）

登录流程：
用户操作 → MainWindow → AuthManager → users.json (验证) → 返回登录状态

数据库操作流程：
用户操作 → MainWindow → SchemaManager (字段校验/表结构管理)
                              ↓
                    StorageManager (创建/删除/修改 .tdf/.trd/.tb 文件)
                              ↓
                    ruanko.log (记录操作日志)

数据操作流程：
用户操作 → MainWindow → RecordManager (增删改查)
                              ↓
                    SchemaManager (数据类型校验/主键唯一性检查)
                              ↓
                    StorageManager (读取.tdf / 读写.trd)
                              ↓
                    {tableName}.trd (二进制数据持久化)
